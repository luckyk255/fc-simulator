## 🤔 什么是"模拟器"?

想象这样一个场景:

> 你有一张1980年代的游戏卡带,但家里只有现代电脑,没有插卡带的红白机。
>
> 模拟器 = 在电脑里"假装"自己是一台红白机,让游戏以为自己真的在红白机上运行

**工作原理(超简化版)**:

```
游戏ROM(.nes文件)
    ↓
模拟器(假装自己是红白机)
    ↓
├─ CPU模块:「我来执行游戏代码」
├─ 显卡模块:「我来画游戏画面」
├─ 声卡模块:「我来播放游戏音效」
└─ 输入模块:「我来接收你的键盘操作」
    ↓
浏览器 Canvas + WebAudio
    ↓
你的屏幕和音响
```

类比说明:

- **模拟器** = 会模仿红白机说话和行为的"翻译官"
- **CPU模块 ** = 翻译官的"逻辑思维中心"
- **显卡模块 ** = 翻译官的"手和画笔"
- **声卡模块** = 翻译官的"嘴巴"
- **输入模块** = 翻译官的"耳朵"

## 🎯 模拟器是怎么工作的?

### Step 1: 读取游戏数据

当你拖拽一个 `.nes` 文件进来时:

- 模拟器读取文件内容(就像你插入游戏卡带)
- 解析ROM的头部信息(了解这是什么游戏、用哪种卡带mapper)
- 加载游戏程序代码到虚拟内存

### Step 2: 执行游戏代码

模拟器的CPU开始工作:

- 一条一条读取游戏指令("马里奥向右走"、"播放跳跃音效"、"检查是否踩到敌人")
- 解释指令并执行(每个指令可能只需要几微秒)
- 红白机每秒执行约180万条指令,模拟器用现代电脑的速度来"假装"这个速度

### Step 3: 画图

60次每秒,PPU(显卡)会做:

- 从VRAM(显存)读取背景Tile(8x8的小方块)
- 像拼图一样把Tile拼成完整背景
- 处理所有精灵(游戏角色、敌人、道具)
- 解决遮挡问题(谁在谁前面)
- 使用调色板给像素上色(NES只有52种颜色)
- 输出256x240的画面

**技术细节**:

```
每帧 = 29780个CPU周期
≈ 166个扫描线
≈ 每扫描线341个点
= 60帧/秒(流畅动画)

就像画家:
- 先画背景(天空、地面、山)
- 再画精灵(马里奥、敌人、金币)
- 最后处理特效
```

### Step 4: 播放声音

APU(声卡)负责:

- 生成方波(大多数音效的音色)
- 生成三角波(低音部分)
- 处理音频混合
- 通过WebAudio API播放

红白机的声音很简单:

```
2个方波通道(旋律)  → 像口哨声
1个三角波通道(低音) → 像嗡嗡声
1个噪声通道(爆炸、枪声) → 像白噪音
```

### Step 5: 接收玩家输入

Joystick(输入模块)会:

- 监听键盘事件(你按下WASD或方向键)
- 把键盘按键转换成红白机手柄信号
- 通知CPU:"玩家按了'A键'"
- CPU让游戏角色做出对应动作

**按键映射**:

```
红白机手柄 → 键盘方案1 → 键盘方案2
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
方向键上    → 键盘↑      → W
方向键下    → 键盘↓      → S
方向键左    → 键盘←      → A
方向键右    → 键盘→      → D
A按钮(跳)   → Z          → K
B按钮(跑)   → X          → J
SELECT      → 数字键2    → G
START       → 数字键1    → H
```

### Step 6: 循环

以上5个步骤每秒重复60次:

```
loop {
  1. CPU执行指令 (3万-6万个周期)
  2. PPU画图 (341点 × 262线)
  3. APU播放声音
  4. 检查玩家输入
  5. 等待下一帧 (保持60fps)
}

就像电影:
→ 每帧画面快速切换
→ 人眼产生"动画"错觉
→ 加上声音 = 完整的游戏体验!
```

## 🛠️ 核心组件详解

### 1. CPU (Central Processing Unit) - 大脑

**作用**: 执行游戏程序的所有指令

**类比**: 就像指挥官,告诉其他部件该做什么

**技术细节**:

```
型号: MOS 6502 (2A03版本)
主频: 1.79 MHz (每秒179万次运算)
指令数: 56条基本指令
寄存器:
  - PC: 程序计数器(当前执行到哪)
  - A: 累加器(做数学题)
  - X, Y: 通用寄存器(辅助计算)
  - S: 栈指针(临时数据堆栈)
  - P: 状态寄存器(记录运算结果状态)

工作方式:
1. 从内存读取指令
2. 解码指令含义
3. 执行指令操作
4. 更新寄存器
5. 重复步骤1-4
```

**常用指令类型**:

```
LDA #$05    → 加载数字5到累加器
STA $2000   → 把累加器的值存到地址$2000
JMP $8000   → 跳转到地址$8000继续执行
CMP #$10    → 比较累加器和数字16
BEQ +5      → 如果相等,向前跳5个字节
```

### 2. PPU (Picture Processing Unit) - 显卡

**作用**: 把数字变成画面

**工作频率**: 5.37 MHz (CPU的3倍,因为画图要快)

**技术细节**:

```
画面规格:
  - 分辨率: 256 × 240 像素
  - 帧率: 60 FPS
  - 颜色: 52种(NTSC制式)
  - 精灵: 最多64个(同时显示最多8个)

内存布局:
  Pattern Table: Tile图案数据(字母、数字、图形)
  Name Table: 背景Tile地图(哪个位置放哪个Tile)
  Attribute Table: Tile颜色和属性
  OAM: 精灵属性内存(角色、敌人位置、颜色等)

渲染流程:
  1. 清空画布
  2. 绘制背景层
  3. 绘制精灵层
  4. 处理精灵优先级
  5. 应用调色板
  6. 输出像素到屏幕
```

**重要概念**:

```
Tile (图块):
 ┌─────┐
 │█████│ 8x8像素的小格子
 │ ███ │ 整屏背景由30×34=1020个Tile组成
 │ ███ │ 红白机内置了512个不同的Tile
 │█████│
 └─────┘

Sprite (精灵):
 游戏角色、敌人、道具等可移动对象
 每个精灵有4个属性:
   - Y坐标 (0-255)
   - Tile编号 (用哪个8x8图案)
   - 属性 (调色板、水平/垂直翻转)
   - X坐标 (0-255)

Palette (调色板):
  背景用4个调色板(每个4色)
  精灵用4个调色板(每个4色)
  实际颜色加起来: NES只有52种颜色

滚动 (Scrolling):
  通过修改内部寄存器实现画面卷轴
  典型的横向/纵向移动游戏效果
```

### 3. APU (Audio Processing Unit) - 声卡

**作用**: 生成游戏音效

**技术细节**:

```
声音通道:
  Pulse Channel 1: 矩形波(主音旋律)
  Pulse Channel 2: 矩形波(和声/音效)
  Triangle Channel: 三角波(低音/打击乐)
  Noise Channel: 噪声(爆炸/枪声)
  DMC Channel: 采样(游戏很少用)

技术参数:
  采样率: 48kHz
  音量: 16级
  占空比: 4种(12.5%, 25%, 50%, 75%)
```

### 4. Memory (内存管理)

**作用**: 存储游戏数据和状态

**内存布局**:

```
CPU内存空间 (64KB):
  $0000-$07FF: 内置RAM (2KB)
  $0800-$0FFF: 内置RAM镜像
  $1000-$17FF: 内置RAM镜像
  $1800-$1FFF: 内置RAM镜像
  $2000-$2007: PPU寄存器
  $2008-$3FFF: PPU寄存器镜像
  $4000-$401F: APU寄存器+控制器
  $4020-$5FFF: 卡带扩展区
  $6000-$7FFF: 卡带SRAM(存档)
  $8000-$FFFF: 卡带ROM(游戏程序)

PPU内存空间 (16KB):
  $0000-$1FFF: Pattern表(图块图案)
  $2000-$23FF: Name表0(背景地图)
  $2400-$27FF: Name表1(背景地图)
  $2800-$2BFF: Name表2(镜像)
  $2C00-$2FFF: Name表3(镜像)
  $3000-$3EFF: $2000-$2EFF镜像
  $3F00-$3F1F: 调色板(背景+精灵)
  $3F20-$3FFF: 调色板镜像
```

**寻址方式**:

```
立即寻址: LDA #$05      (数字5就是值)
直接寻址: LDA $2000     (地址$2000的内容)
间接寻址: LDA ($20),Y   (先读$20地址的值,再加Y)
```

### 5. Mapper (卡带类型适配器)

**作用**: 不同的游戏卡带有不同的内存映射方式,Mapper负责适配

**常见类型**:

```
Mapper 0: 最简单,32KB ROM
Mapper 1: MMC1芯片,可以切换内存页
Mapper 2: UNROM,最大64KB ROM
Mapper 3: CNROM,可以切换图案表

工作原理:
  就像游戏卡带的"翻译官"
  CPU说:"读取地址$8000"
  Mapper决定:"这是卡带ROM的第几块"
  返回正确的数据给CPU
```

### 6. Joystick (手柄输入)

**作用**: 把键盘按键转换成红白机手柄信号

**实现**:

```
原理: 手柄是1字节(8位)的串行输入
位定义:
  bit 0: A按钮
  bit 1: B按钮
  bit 2: SELECT
  bit 3: START
  bit 4: 方向上
  bit 5: 方向下
  bit 6: 方向左
  bit 7: 方向右

工作流程:
  1. 监听键盘keydown/keyup事件
  2. 映射到对应手柄位
  3. CPU读取$4016地址
  4. 返回当前按键状态
```

## 🎨 进阶知识: 游戏画面怎么画出来的?

### 游戏画面 = 背景 + 精灵

```
┌─────────────────────────┐
│  PPU绘制顺序            │
│                         │
│  1. 清空画布            │
│     #000000 (黑色)      │
│                         │
│  2. 画背景层            │
│     从Name Table读Tile  │
│     32×30=960个背景Tile │
│     应用背景调色板      │
│                         │
│  3. 画精灵层            │
│     从OAM读精灵属性     │
│     最多64个精灵        │
│     应用精灵调色板      │
│                         │
│  4. 混合叠加            │
│     精灵在背景前面      │
│     透明色=背景显示     │
│     处理优先级          │
│                         │
│  5. 应用效果            │
│     灰度、颜色强调      │
│                         │
│  6. 输出到屏幕          │
│     256×240像素图像     │
└─────────────────────────┘
```

### Tile (图块)系统详解

游戏画面用小块拼成:

```
┌─────────────┐
│ Pattern Table│ Tile图案数据库
├─────────────┤
│ 000: █ ████ │ 每个数字对应一个8×8的图案
│ 001: ███ █  │ 比如0是云的左上角
│ 002:  ████  │ 1是云的右上角
│ 003: ██████ │ 2是树干
│ 004:   ██   │ 3是树叶
│ ...          │ ...
│ 511: ██████ │ 总共512个Tile
└─────────────┘

┌─────────────┐
│ Name Table   │ Tile摆放地图
├─────────────┤
│ 0,0: 001    │ "左上角放Tile 001(云朵右上)"
│ 0,1: 002    │ "往右一格放Tile 002(云朵中上)"
│ ...         │
│ 31,29: 123  │ "右下角放Tile 123(地面)"
│ 32×30 Grid  │ 30行×32列=960格
└─────────────┘
```

### 精灵 (Sprite)系统

会动的游戏元素:

```
┌─────────────────┐
│ OAM (精灵属性)  │
├─────────────────┤
│ Sprite 0:       │
│   Y:  100       │ 屏幕位置Y
│   Tile: 24      │ 图案编号
│   Attr: $47     │ 调色板(高2位)
│                  │   水平翻转(bit6)
│                  │   垂直翻转(bit7)
│   X:  150       │ 屏幕位置X
│                  │
│ Sprite 1:       │
│   Y:  105       │
│   Tile: 25      │
│   Attr: $47     │
│   X:  150       │
│ ...             │ 总共64个精灵槽位
└─────────────────┘

马里奥=4个Sprite拼成:
┌───┬───┐
│24 │25 │ 头顶
├───┼───┤
│26 │27 │ 身体
└───┴───┘
```

### 调色板 (Palette)

NES只有52种颜色:

```
┌─────────────────────────────────┐
│ 背景调色板(4组,每组4色)         │
├─────────────────────────────────┤
│ 第0组:                          │
│   0: 透明(显示底色)            │
│   1: #3CBCFC (天空蓝)          │
│   2: #0078F8 (深蓝)            │
│   3: #000000 (黑)              │
│                                 │
│ 第1组: 地面/砖块               │
│ 第2组: 敌人/道具               │
│ 第3组: 水管/特殊物品           │
└─────────────────────────────────┘

精灵调色板也4组,但第一色强制透明
总共: 4×4=16背景色 + 4×3=12精灵色
实际上: NES硬件只支持52种颜色
```

## ❓ 常见问题

### Q1: 这是什么?能玩什么游戏?

**A**: 这是一个红白机游戏模拟器,可以玩:
- 超级马里奥兄弟
- 魂斗罗
- 坦克大战
- 冒险岛
- 忍者龙剑传
- 1942
- 以及其他用Mapper 0/1/2/3的游戏(约400+款)

### Q2: 为什么能模拟红白机?

**A**: 红白机的硬件:
- 非常简单(1980年代的技术)
- 文档齐全(逆向工程几十年了)
- 速度不快(1.79MHz CPU)
- 现代电脑完全有能力"假装"自己是红白机

### Q3: 需要联网吗?

**A**: 完全不需要!

- 纯HTML+JavaScript实现
- 所有文件在本地运行
- 没有服务器,没有API调用
- 断网也能玩(把文件下载到本地即可)

### Q4: 支持手机玩吗?

**A**: 支持!

- 响应式设计,自动适应屏幕
- 触摸屏支持(左右虚拟按键)
- 推荐用蓝牙键盘,体验更好

### Q5: 为什么画面模糊?

**A**: 解决办法:

```
1. 点击【原始画质】按钮
2. 或者按键盘【P】键切换
3. 取消图像平滑处理(get crispy像素)
```

### Q6: 怎么存档?

**A**: 提供两种存档方式:

**方式1 - 游戏内存档**: 支持使用游戏自带的存档功能(比如超级马里奥的中途保存)

**方式2 - 状态保存**: 按【F5】保存当前状态到浏览器缓存,按【F9】读取

### Q7: 如何退出游戏?

**A**: 两种方法:

- 刷新浏览器页面(F5)
- 拖拽一个新的游戏ROM进来切换

### Q8: 声音为什么有杂音?

**A**: 可能原因:

1. **性能不够**: 降低窗口大小,减少CPU负担
2. **浏览器问题**: 尝试Chrome或Edge
3. **音频延迟**: 在设置里调整音频缓冲区(Cache size)

### Q9: 如何自己开发游戏?

**A**: 推荐工具:

```
游戏开发:
  - FamiStudio (可视化音乐编辑器)
  - NESMaker (可视化游戏制作)
  - asm6/ca65 (6502汇编编译器)

ROM测试:
  - Mesen (推荐,开源)
  - FCEUX (老牌模拟器)
  - 本模拟器可以直接加载测试
```

## ⚡ 技术架构总结

### 整体架构

```
machine.js            (总指挥: 协调所有模块)
    ├── cpu.js        (大脑: 解释游戏代码)
    ├── ppu.js        (画家: 绘制游戏画面)
    ├── apu.js        (歌手: 播放游戏音效)
    ├── memory.js     (仓库: 存储数据)
    ├── mapper.js     (翻译: 适配不同游戏)
    └── joystick.js   (耳朵: 接收玩家输入)

main.js               (主程序: 连接DOM界面)
index.html            (网页: 显示Canvas)
```

### 数据流

```
1. 加载ROM文件
   ↓
2. 解析iNES格式 (ines.js)
   ↓
3. 初始化Mapper
   ↓
4. CPU重置 (执行RESET中断)
   ↓
5. 进入主循环 (60fps)
   ├→ CPU执行指令
   ├→ PPU渲染画面
   ├→ APU播放音频
   └→ 检查用户输入
   ↓
6. 输出到Canvas + WebAudio
```

### 关键算法

**CPU周期计数**:
- 每条指令有固定的执行周期(cycles)
- CPU.cycleCount累加这些cycles
- 达到29780 cycles = 一帧完成
- 触发NMI,开始下一帧

**PPU渲染定时**:
- 341 dots × 262 scanlines = 89342 PPU cycles
- 每帧 = 29780 CPU cycles (因为PPU是CPU速度的3倍)
- CPU通过2002端口查询VBLANK状态
- 游戏利用VBLANK期间更新画面数据

**Sprite优先级**:
- Sprite表中序号小的优先(0号Sprite最高)
- 背景可以在Sprite前面(如果Sprite属性设置)
- Y坐标小的在上方
